# US-005 — Security and observability

## Status

Planned

## Summary

As a platform engineer,
I want every request to the gateway to be authenticated (via OAuth2 JWT or API key) and every transaction to produce a Prometheus metric and a distributed trace,
so that only authorized clients can reach downstream services and the operations team can observe latency, error rates, and request flows without reading raw logs.

## Business / Engineering value

Prior to this story, the gateway is fully open — anyone with network access can reach downstream services. Correlation IDs and request logs exist, but they are not queryable at scale and cannot be correlated across services.

This story closes both gaps:

- **Security**: Spring Security WebFlux resource server validates JWT Bearer tokens issued by an external IdP. API keys generated by `hermes-admin` are validated via Redis (written by admin on creation, read by the gateway on every request). Unauthenticated requests are rejected with `401 Unauthorized` before reaching any filter or downstream.
- **Observability**: Micrometer + Prometheus exposes gateway metrics (request count, latency, rate-limit rejections, circuit breaker state) at `/actuator/prometheus`. OpenTelemetry propagates trace context across the gateway and downstream services using W3C `traceparent` headers. Trace IDs are correlated with the existing `X-Correlation-ID` mechanism.

Together, these capabilities make `hermes-gateway` production-observable and production-secure.

## Scope

This story includes:

- `spring-boot-starter-oauth2-resource-server` configured in `hermes-gateway` for JWT Bearer token validation.
- `SecurityWebFilterChain` bean: permit `/actuator/health`, `/actuator/prometheus`, `/fallback`; require authentication everywhere else.
- An `ApiKeyAuthFilter` (`GlobalFilter`) that validates `X-API-Key` headers against Redis (hashes written by `hermes-admin`) and injects a `SecurityContext` entry when valid.
- `hermes-admin` extended to publish API key hashes to Redis on creation and deletion.
- `micrometer-registry-prometheus` in `hermes-gateway`; `/actuator/prometheus` exposed.
- Custom Micrometer counters: `hermes.rate.limit.rejected` and `hermes.circuit.breaker.open`.
- `micrometer-tracing-bridge-otel` + OTLP exporter configured in `hermes-gateway`.
- W3C `traceparent` propagation in outbound requests; trace ID correlated with `X-Correlation-ID`.

## Out of scope

This story does not include:

- Grafana dashboard provisioning (deferred — Prometheus endpoint is the deliverable; dashboards are ops tooling).
- AWS X-Ray integration (deferred to v1.0.0 production deployment story).
- CloudWatch Logs structured export.
- OAuth2 Authorization Server setup (the gateway is a resource server only — it validates tokens, not issues them).
- Role-based authorization (RBAC) — authentication only in this milestone.
- HTTPS/TLS termination (handled by the ALB/Ingress in production; not relevant for local development).
- Admin API (`hermes-admin`) authentication (deferred — the admin service is internal network only).

## Acceptance criteria

- [ ] A request without `Authorization` or `X-API-Key` header receives `401 Unauthorized`.
- [ ] A request with a valid JWT Bearer token is forwarded to the downstream service.
- [ ] A request with a valid `X-API-Key` is forwarded; an invalid or revoked key receives `401`.
- [ ] `GET /actuator/prometheus` returns Prometheus-format metrics including `http_server_requests_seconds_count`.
- [ ] Custom counter `hermes_rate_limit_rejected_total` increments when a request is rate-limited.
- [ ] Each request produces a trace with a `traceparent` header forwarded to the downstream service.
- [ ] The trace ID is reflected in (or identical to) the `X-Correlation-ID` response header.
- [ ] `.\mvnw.cmd clean verify` passes from the repository root.

## Technical notes

- **Reactive security**: Spring Security WebFlux uses `SecurityWebFilterChain` (not `HttpSecurity`). Annotate the config class with `@EnableWebFluxSecurity`. Never use `HttpSecurity` or `WebSecurityConfigurerAdapter` — those are Spring MVC only.
- **JWT issuer**: configure `spring.security.oauth2.resourceserver.jwt.issuer-uri` in `application.yml`. For local development, use a local Keycloak container or a test JWT with a known secret and `spring.security.oauth2.resourceserver.jwt.secret` (HS256 for local only — RS256 in production).
- **API key validation flow**: `ApiKeyAuthFilter` reads `X-API-Key` from the request header → computes `SHA-256(rawKey)` → checks Redis key `apikey:{hash}` → if found and active, creates a `UsernamePasswordAuthenticationToken` with the tenant ID as principal and sets it in the `ReactiveSecurityContextHolder`.
- **Redis key schema for API keys**: `apikey:{keyHash}` → JSON value `{"tenantId":"...", "active":true, "label":"..."}`. Written by `hermes-admin` via `ReactiveRedisTemplate` when a key is created; deleted when a key is revoked.
- **Micrometer**: already on the classpath via `spring-boot-starter-actuator`. Only `micrometer-registry-prometheus` needs to be added.
- **Tracing**: `micrometer-tracing-bridge-otel` bridges Micrometer's `Tracer` API to OpenTelemetry. The OTLP exporter sends spans to a local collector (Zipkin-compatible or OTLP endpoint). For local dev, configure the exporter to log to console (`logging` exporter) to avoid requiring a running collector.
- **`X-Correlation-ID` ↔ trace ID alignment**: update `CorrelationIdFilter` to use the current trace ID (from `Tracer.currentSpan().context().traceId()`) as the correlation ID when a trace is active, falling back to UUID generation when no trace context exists.

```bash
# Obtain a token from the local IdP, then:
curl -s -H "Authorization: Bearer $TOKEN" http://localhost:8080/api/echo/ping
```

API key smoke test:

```bash
RAW_KEY=$(curl -s -X POST http://localhost:8081/admin/api-keys \
  -H "Content-Type: application/json" \
  -d '{"label":"test","tenantId":"team-platform"}' | jq -r .rawKey)

curl -s -H "X-API-Key: $RAW_KEY" http://localhost:8080/api/echo/ping
```

Prometheus check:

```bash
curl -s http://localhost:8080/actuator/prometheus | grep hermes_rate_limit
```

## Notes

- The gateway acts as an OAuth2 resource server — it validates tokens but never issues them. The authorization server (Keycloak, Auth0, AWS Cognito, etc.) is external to this project.
- For local development without a real IdP, configure `spring.security.oauth2.resourceserver.jwt.public-key-location` with a locally generated RSA key pair, or use a test-only Keycloak container added to `docker-compose.yml`.
- API key validation adds one Redis read per request on the hot path. Redis round-trip latency on `localhost` is sub-millisecond — this is acceptable. In production (ElastiCache), latency is 1–3 ms.
