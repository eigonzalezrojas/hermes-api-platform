# US-004 — Admin API

## Status

Planned

## Summary

As a platform engineer,
I want a REST API that lets me create, read, update, and delete gateway routes, tenants, and API keys without restarting or redeploying the gateway,
so that the platform can be configured dynamically and multiple teams can be onboarded as tenants with their own isolated rate limits and access credentials.

## Business / Engineering value

Without an admin API, every routing change requires a code edit, a commit, a build, and a deployment. For an API gateway serving multiple teams, this is operationally untenable.

This story introduces the `hermes-admin` service — a separate Spring Boot application with its own PostgreSQL database. It exposes a management REST API covering the three core domain objects:

- **Routes**: the routing rules the gateway applies to incoming traffic.
- **Tenants**: organizational units that own routes and API keys, enabling multi-tenancy isolation.
- **API Keys**: per-tenant credentials used to authenticate and rate-limit downstream callers.

Separating admin concerns into `hermes-admin` is an intentional security boundary: the gateway (`hermes-gateway`) has no database connection and holds no credentials. Only `hermes-admin` talks to PostgreSQL.

## Scope

This story includes:

- The `hermes-admin` Maven module: Spring Boot 3.x with Spring MVC, Spring Data JPA, Flyway, and PostgreSQL.
- PostgreSQL service added to `docker-compose.yml`.
- Three Flyway migrations: `tenants`, `routes`, and `api_keys` tables.
- REST CRUD endpoints for routes (`/admin/routes`), tenants (`/admin/tenants`), and API keys (`/admin/api-keys`).
- API key generation: plaintext key returned once on creation, only the hash stored in the database.
- Bean Validation on all request DTOs.
- Repository-layer tests with Spring Data JPA test slice (`@DataJpaTest`).
- Controller-layer tests with MockMvc test slice (`@WebMvcTest`).

## Out of scope

This story does not include:

- **Dynamic route reload in the gateway**: the gateway reads its routes from `application.yml` in this milestone. Integration between `hermes-admin` and `hermes-gateway` (so that admin changes propagate at runtime) is deferred to v1.0.0.
- Authentication on the admin API (deferred to the security story, v0.5.0).
- Pagination on list endpoints (deferred — acceptable for an admin API with low cardinality).
- Soft-delete (hard delete is used throughout).
- API key rotation (a dedicated endpoint is deferred; deletion + re-creation covers the MVP).
- Multi-tenancy enforcement in `hermes-gateway` (deferred to v0.5.0).

## Acceptance criteria

- [ ] `docker compose up -d` starts both Redis and PostgreSQL containers without errors.
- [ ] Flyway applies all migrations on `hermes-admin` startup (`V1`, `V2`, `V3`).
- [ ] `POST /admin/routes` creates a route and returns `201 Created` with the created resource.
- [ ] `GET /admin/routes` returns all routes; `GET /admin/routes/{id}` returns a single route.
- [ ] `PUT /admin/routes/{id}` updates a route; `DELETE /admin/routes/{id}` deletes it.
- [ ] The same CRUD contract applies to `/admin/tenants` and `/admin/api-keys`.
- [ ] `POST /admin/api-keys` returns the plaintext key in the response body exactly once.
- [ ] `GET /admin/api-keys/{id}` does NOT return the plaintext key or its hash.
- [ ] Invalid requests (missing required fields) return `400 Bad Request` with a validation error body.
- [ ] `.\mvnw.cmd clean verify` passes from the repository root.

## Technical notes

- **hermes-admin** is a Spring MVC application (not WebFlux). Spring Data JPA is incompatible with WebFlux — JPA is blocking by design. The admin API has low traffic and no reactive requirements.
- **Port**: `8081`. The gateway (`hermes-gateway`) runs on `8080`. Both can run simultaneously.
- **Base package**: `dev.eithel.hermes.admin`.
- **Database**: PostgreSQL 16. Connection via `spring.datasource.url=jdbc:postgresql://localhost:5432/hermesdb`. JDBC driver: `org.postgresql:postgresql`.
- **ORM**: Spring Data JPA with Hibernate. Primary keys are `UUID` generated by the database (`@GeneratedValue(strategy = GenerationType.UUID)` — Java 21 / Hibernate 6 compatible).
- **Flyway**: auto-runs on startup. Migrations in `src/main/resources/db/migration/`. Naming: `V{n}__{description}.sql`.
- **Predicates and filters** on `Route` entity: stored as `TEXT` (JSON-serialized string). A JPA `AttributeConverter` handles `List<String> ↔ TEXT` serialization via Jackson `ObjectMapper`. This is simpler than `@ElementCollection` for the MVP.
- **API key hashing**: `SHA-256` with a random salt prefix (`UUID`). The full value stored is `{salt}:{hex(SHA-256(salt + rawKey))}`. BCrypt is avoided because it is slow by design and the admin API is low-frequency — SHA-256 is sufficient for long random keys.
- **Security boundary**: `hermes-admin` has no inbound route in `hermes-gateway`. It is accessible only via internal network or VPN, not through the public gateway.

## Validation

Start infrastructure:

```bash
docker compose up -d
```

Start admin service:

```powershell
.\mvnw.cmd -pl hermes-admin spring-boot:run -Dspring-boot.run.profiles=local
```

Smoke test:

```bash
# Create a tenant
curl -s -X POST http://localhost:8081/admin/tenants \
  -H "Content-Type: application/json" \
  -d '{"name": "team-platform", "tenantId": "platform"}' | jq .

# Create an API key for the tenant
curl -s -X POST http://localhost:8081/admin/api-keys \
  -H "Content-Type: application/json" \
  -d '{"label": "ci-token", "tenantId": "platform"}' | jq .
```

Full build:

```powershell
.\mvnw.cmd clean verify
```

## Notes

- The `hermes-admin` module is a standalone Spring Boot application — it has its own `main` class and its own `application.yml`. It does not inherit the gateway's configuration.
- The dynamic gateway reload (publishing route changes from admin to gateway at runtime) requires an event bus or a shared `RouteDefinitionRepository` backed by PostgreSQL. This is the key integration work deferred to v1.0.0. The admin API stores routes in the DB now so that the data model is in place when the integration is built.
